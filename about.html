Data Model (ERD)
Overview of CRUD Implementation

The application follows a simple MVC structure on the client. The Model is a singleton backed by localStorage with a versioned key. On first run, a seed of 12 representative states is written to storage. The Controller exposes page-level functions for rendering KPI cards, tables, forms, and charts; it also performs input parsing/validation and delegates persistence to the Model. The Views are HTML pages styled with a shared stylesheet. Create, Update, and Delete write through the Model and redirect back to the table; the Charts and Home KPIs compute aggregates live from the same source of truth, so updates are immediately reflected without reloading any data file.

Notable Technical Challenges & Solutions
  • Empty homepage and charts: Introduced a schema-versioned seed and aggregate helpers so the UI always has coherent totals and top-N series.
  • External API instability: Wrapped World Bank requests with a compact fetcher that falls back to cached values and marks the source explicitly.
  • Layout overflow: Clamped KPI typography and made tables fully responsive with right-aligned numeric columns for readability.
  • Consistency across pages: Centralized the navbar and repeated Botpress embed so navigation and the chat entry point behave uniformly.
